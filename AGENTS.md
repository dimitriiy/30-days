 AGENTS.md — React Frontend

Ты — ИИ‑разработчик в современном React‑проекте.  
Цель: писать чистый, поддерживаемый код с атомарными компонентами и явной инъекцией зависимостей (DI), делая **минимальные** точечные изменения.

## 1. Стек и контекст

- Язык: TypeScript strict.
- UI: React 18+ с функциональными компонентами и хуками, NEXTJS.
- Роутинг: React Router / Next.js App Router (ориентируйся по существующему коду).
- Стили: Tailwind  
- Состояние: React hooks + Reatom JS

Если в проекте уже есть `STYLEGUIDE.md` или `design-system` — следуй именно им, не изобретай свой стиль.


Правила:

- Новый UI‑элемент сначала ищи в существующих атомах/молекулах — не дублируй компоненты.
- Не добавляй бизнес‑логику в атомы: вся логика — выше по иерархии (molecules/organisms/pages).
- Следи за размером компонентов: если компонент стал «толстым» (много JSX или логики) — предложи разбить его на более мелкие.

## 3. Инъекция зависимостей (DI) и разделение слоёв

Разделяем UI, доменную логику и инфраструктуру.

- Компоненты не должны напрямую знать о реализации API/хранилища — только о абстракциях.
- Работа с API, storage, feature‑флагами, логированием и т.п. инкапсулируется в сервисах/адаптерах.
- DI реализуем через:
  - пропсы (передача функций/сервисов в компоненты),
  - контекст (React Context) для верхнеуровневых зависимостей,
  - кастомные хуки, инкапсулирующие доступ к сервисам (`useAuth()`, `useUserRepo()`, `useFeatureFlags()`).

Правила:

- В компонентах верхнего уровня (pages/containers) можно связывать сервисы и UI.
- В атомах/молекулах запрещены прямые вызовы API, глобальных синглтонов и `window.localStorage`.
- Вместо `fetch` внутри компонента используй существующие абстракции (например, `apiClient`, `userRepository`, `useQuery` хуки).
- Если нужного слоя абстракции нет, предложи создать его (интерфейс + реализация) и опиши, куда положишь эти файлы.

## 4. Clean Code для React

Общие принципы:

- Функциональные компоненты с хуками, без классов.
- Один компонент — одна ответственность. Если компонент делает сразу несколько разных вещей — раздели.
- Чёткие границы: UI‑компоненты не осуществляют бизнес‑решений, сервисы не знают о JSX.

Стиль:

- Понятные имена: `UserProfileCard`, `useUserProfile`, `UserRepository`, `AuthContext`.
- Минимизируй вложенность JSX: выноси поддеревья в отдельные компоненты.
- Не дублируй код. Если кусок повторяется 3+ раза — вынеси в компонент/хук.
- Не используй `any` без крайней необходимости; вывод типов допускается, но публичные границы (props, возвращаемые значения) должны быть типизированы явно.

Условный рендеринг и эффекты:

- Избегай сложных вложенных тернарных операторов — используй ранние return‑ы или вспомогательные переменные.
- Эффекты (`useEffect`) должны быть маленькими и осмысленными. Не пихай в один эффект кучу не связанных операций.

## 5. Работа с состоянием

- Локальное состояние (`useState`, `useReducer`) для UI‑деталей (модалки, инпуты, локальные фильтры).
- Глобальное состояние — только через уже принятый в проекте стейт‑менеджер.
- Не дублируй источник правды: данные либо в кэше клиента (query‑клиент, store), либо в локальном состоянии; избегай зеркалирования без нужды.

Правила:

- Избегай «проброса пропсов через пять уровней» — используй контекст или композицию.
- Не храни в состоянии вычисляемые значения — вычисляй на лету, если это недорого или мемоизируй (`useMemo`), если дорого.
- Не храни в состоянии целые сущности, если достаточно ID.



## 7. Стиль кода и форматирование

- Используй существующий форматтер (Prettier, ESLint) и не форматируй файлы целиком, если правка точечная.
- Следуй общему стайлу: отступы, длина строки, кавычки, запятые — смотри в конфиг проекта.
- Импорты:
  - Сначала стандартные библиотеки/React,
  - затем внешние пакеты,
  - затем внутренние модули (alias/relative).
- Удаляй неиспользуемые импорты и переменные.

## 8. Что можно и что нельзя делать

### Можно

- Вносить минимальные изменения, необходимые для решения задачи.
- Выделять новые атомы/молекулы/хуки, если это уменьшает дублирование и делает код яснее.
- Улучшать типизацию и обработку ошибок в изменяемых областях.

### Нельзя

- Переформатировать большие файлы без необходимости.
- Менять публичные API компонентов/хуков без явной причины.
- Добавлять новые крупные зависимости без указания и явной просьбы.
- Встраивать бизнес‑логику в атомы и низкоуровневые компоненты.

## 9. Как выполнять задачи

При любой задаче:

1. Прочитай контекст: связанные компоненты, хуки, сервисы, AGENTS.md и при наличии STYLEGUIDE.md.
2. Составь краткий план решения (2–5 шагов) и следуй ему.
3. Делай минимальный дифф: измени только то, что нужно.
4. Обнови/добавь тесты.
5. Проверь линтер и типы.

Если что‑то непонятно (архитектура, выбор слоя, API):

- Сформулируй один‑два уточняющих вопроса **и** предложи разумный план, как поступить в каждом варианте.
